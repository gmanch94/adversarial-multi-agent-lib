_None._

## HIGH

### A8-H-01 — docker-compose scheduler DSN has no password; postgres rejects unauth → either silent connect failure or auth bypass via `pg_hba` trust default
**File:** `examples/production/durable_postgres/docker-compose.yml`, line 40
**Files referenced:** lines 17–25 (postgres service block), line 37 (`env_file: .env`)

```yaml
scheduler:
    env_file: .env                                              # POSTGRES_DSN with password from .env
    environment:
      POSTGRES_DSN: "postgresql://daemon@postgres:5432/durable" # OVERRIDES env_file. No password.
```

Compose precedence: `environment:` block wins over `env_file:`. So the operator's carefully-set `.env` value (with the password the README told them to set) is **silently discarded** by the compose override.

Two outcomes, both bad:
1. **Happy path on first deploy:** `postgres:16-alpine` ships with `pg_hba.conf` defaulting to `scram-sha-256` for non-local hosts → daemon's `asyncpg.create_pool()` raises `InvalidPasswordError` at startup → container restart-loops → operator debugs for hours.
2. **If operator "fixes" it by relaxing `pg_hba`** to `trust` for the internal network (predictable response when "it just won't connect" + the README says "internal docker network is the trust boundary at this scope"): any compromised co-resident container on `internal` network → full DB write. The `cap_drop:[ALL]` + `read_only:true` hardening becomes worthless because the auth model collapses.

The README explicitly states "Internal docker network is the trust boundary at this scope" (line 247) — this normalises the trust-auth path.

**Attack vector:** any adversary who lands code execution on the `postgres` or `scheduler` container (e.g. via a future supply-chain compromise of the library, or by adding a malicious helper service to compose) can `psql -h postgres -U daemon durable` with no password from the same internal network, read every encrypted checkpoint, delete rows, install triggers (CHECK constraint doesn't bar UPDATE-anywhere; `daemon_user` has full DML per `schema.sql` comment).

**Fix:**
- Put password into the DSN via init script, e.g. `POSTGRES_DSN: "postgresql://daemon:$$(cat /run/secrets/postgres_password)@postgres:5432/durable"` — but that requires shell exec.
- Better: drop the `environment.POSTGRES_DSN` override entirely; force operator to set the password in `.env`. Document that the example compose talks to `postgres` hostname on internal network.
- Add a `pg_hba.conf` overlay mount with explicit `scram-sha-256` for the daemon role.
- Smoke-test #11 (`test_11b_daemon_logs_clean_of_secrets`) cannot catch this — the DSN it scans for has no password.

### A8-H-02 — `requirements.txt` is empty (placeholder only); Dockerfile `--require-hashes` build will fail
**File:** `examples/production/durable_postgres/requirements.txt`, lines 1–12 (entire file)
**Files referenced:** `Dockerfile` line 25, `requirements.in`

```
# AUTO-GENERATED by pip-compile from requirements.in.
# ... IMPORTANT: this placeholder is filled by a build-time step before
# `docker compose build` is run.
```

No package lines, no hashes. `pip install --require-hashes -r /tmp/requirements.txt` against an empty hashed lockfile installs nothing.

**Impact:** anyone who clones, follows README quickstart steps 1–7 in order, will hit `docker compose build` failures. Plan reviewer v2 noted F-C-03 (setuptools+wheel for build backend) was "verified" — but that verification was against `requirements.in`, not the shipped `requirements.txt`. The lockfile that ships isn't a lockfile.

Plan-time review missed because reviewer assumed pip-compile would be run as part of ship. README step 2 makes that explicit ("first-time only") but the README is just a doc — the file in the repo is incoherent. Worse, when an operator does run `pip-compile`, the hashes pin the *current upstream* of asyncpg / cryptography / bandit / pip-audit / cyclonedx-bom — not the versions the team tested against. So every operator gets a *different* lockfile.

**Fix:** generate the actual hashed lockfile in CI (or once locally) and commit it. The point of `--generate-hashes` is reproducibility; an unfilled file gives the opposite.

### A8-H-03 — Dockerfile base image digest is unfilled placeholder; build fails or operator silently disables pin
**File:** `examples/production/durable_postgres/Dockerfile`, line 9

```dockerfile
FROM python:3.11-slim@sha256:REPLACE_WITH_CURRENT_DIGEST_AT_BUILD_TIME
```

`docker build` will fail with `invalid reference format` on the literal `REPLACE_WITH_...` string. README step 3 tells the operator to fill it manually. Risk: operator under deadline pressure replaces `@sha256:...` with `:3.11-slim` (no pin) just to make it build, defeating the supply-chain control. No tooling enforces the pinned-by-digest invariant.

**Impact:** silent regression from "pinned base image" to "rolling tag." Future Python 3.11-slim base updates ship unscanned to every operator who took the shortcut.

**Fix:**
- Ship with a *current* known-good digest at the time of commit; document the refresh cadence (the file already mentions quarterly).
- Add a `scripts/check_base_image_pinned.sh` CI gate: `grep -E 'FROM python:.*@sha256:[a-f0-9]{64}$' Dockerfile`.

### A8-H-04 — Healthcheck server has no listener-level connection cap; SYN/slow-loris flood from co-resident container can DoS daemon
**File:** `examples/production/durable_postgres/daemon.py`, lines 148–152

```python
self._server = await asyncio.start_server(
    self._handle, host="127.0.0.1", port=self._port
)
```

`asyncio.start_server` accepts the OS-default `SOMAXCONN` (4096 on modern Linux) — no `backlog=` argument, no per-task semaphore. Each connection spawns a `_handle` task that holds for up to `_REQUEST_TIMEOUT_SECONDS = 5.0` (good — N-H-01 closed) before being cleaned up.

The healthcheck binds `127.0.0.1` *inside the container* — so only co-located processes can hit it. With `cap_drop:[ALL]` and `read_only:true`, exploit difficulty is high. But:
- Anyone in the scheduler container's PID namespace (an attached `docker exec` shell from a compromised orchestrator) → free DoS.
- A misconfigured `docker compose ports` exposing 8080 to host or LAN (the README never explicitly warns "never expose 8080") would multiply the impact.
- Worse: docker's `HEALTHCHECK` `urllib.request.urlopen` doesn't set a timeout (compose line 53), so a stuck healthcheck blocks the entire container's health state.

**Impact:** 5000 concurrent slow-loris connections × 5s timeout = 25K task-seconds; even one DoS minute consumes ~5K active asyncio tasks, exhausting event-loop scheduling and starving the actual workflow scheduler.

**Fix:**
- `asyncio.start_server(..., backlog=16)` — healthcheck never needs more.
- Wrap `_handle` in an `asyncio.Semaphore(8)` — concurrent request cap.
- Compose healthcheck `urllib.request.urlopen(url, timeout=3)`.

### A8-H-05 — `_PgLockHandle` mutable + `unsafe_hash=True` → set membership in `_ActiveLockRegistry` is broken if any field mutates
**File:** `examples/production/durable_postgres/lock.py`, lines 52–66, 281, 287

```python
@dataclass(unsafe_hash=True)
class _PgLockHandle:
    ...
    watchdog: Optional[asyncio.Task] = field(default=None)
    released: bool = field(default=False)
```

`unsafe_hash=True` derives `__hash__` from **all fields** including the mutable `watchdog` task ref and `released` bool. The registry uses `set[_PgLockHandle]` (line 281). `register(handle)` hashes the handle at one state (watchdog=None? or watchdog=Task#1?). `unregister(handle)` later hashes at a different state (watchdog=None, released=True after release()). The `set.discard()` and `set.add()` see *different hash keys* for the same logical handle.

Re-reading `release()`:
```python
handle.released = True            # mutates AFTER unregister
if self._registry is not None:
    self._registry.unregister(handle)   # ← lines 179-180 actually unregister FIRST
```

Lines 179–180 unregister *before* mutating `released = True`. Good. But:
```python
handle.watchdog = None            # line 184, after unregister already done
```

This is post-unregister, so the registry membership is consistent at unregister time. **But heartbeat() does the opposite:**
```python
# Line 217
handle.watchdog = None            # mutates watchdog
# ... line 232
handle.watchdog = asyncio.create_task(...)  # mutates again
# Registry never re-registers; if heartbeat() is called, the handle's hash
# is different from when it was registered. force_close_all() iterates
# `list(self._handles)` (line 290) which still works (list copy stable), but
# `self._handles.discard(handle)` at line 298 silently fails because the
# computed hash doesn't match the stored entry.
```

**Result:** `_ActiveLockRegistry._handles` accumulates "ghost" entries that `force_close_all` cannot discard. After many heartbeats, the registry leaks memory and the `for handle in list(...)` loop double-closes already-closed connections.

**Concretely:**
- `register(h)` → `_handles.add(h)` → bucket at `hash(state_A)`
- `heartbeat(h)` mutates `h.watchdog` → next `hash(h) = state_B`
- `unregister(h)` → `_handles.discard(h)` → looks up bucket at `state_B` → no match → discard is no-op
- registry leaks handle even after release

**Impact:** memory growth over time on long-running deployment + SIGTERM `force_close_all` walks stale handles → multiple `await handle.conn.close()` on the same closed connection (exception swallowed, but timing-wise stretches shutdown).

**Fix:**
- Use `eq=False` (identity-based hash) on the dataclass: `@dataclass(eq=False)`. Then `hash(handle) = id(handle)`, immune to field mutation.
- OR convert `_handles` from `set` to `dict[int, _PgLockHandle]` keyed by `id(handle)`.

Spec called for `_PgLockHandle plain dataclass not frozen-base inheritance` — that part is fine; the hash strategy is the bug.

### A8-H-06 — `reencrypt_all` reaches through `EncryptedCheckpointStore._inner` AND `._encrypt_request_json` private API; brittle to library refactor + bypasses sentinel idempotency check
**File:** `examples/production/durable_postgres/scripts/reencrypt_all.py`, lines 57, 79

```python
inner: PostgresCheckpointStore = store._inner
re_encrypted = store._encrypt_request_json(cp)
```

Two issues:

1. **Private-API coupling.** Both `_inner` and `_encrypt_request_json` are underscore-prefixed. A library version bump (rename, refactor to property, hide behind `_PRIVATE_API_`) silently breaks rotation. Rotation is a security-critical path. The plan reviewer noted "reaching THROUGH" in a docstring but accepted it.

2. **Sentinel-already-encrypted skip.** `_encrypt_request_json` (encryption.py line 48):
```python
if cp.last_request_json.startswith(self._ENC_PREFIX):
    return cp  # Already encrypted, pass through
```
But `store.read(run_id)` (line 76 of reencrypt_all) decrypts first → `cp.last_request_json` is plaintext when fed to `_encrypt_request_json`. So this works *today*. But if a future library change short-circuits the decrypt (legacy passthrough on missing prefix per encryption.py line 70) and emits the plaintext through, `_encrypt_request_json` re-encrypts plaintext-with-prefix — never breaks the rotation but never *completes* it either. The "Re-encrypted N rows" output lies.

**Impact:** key rotation is the canonical compromise-response path (README line 102: "Compromise response"). A silent rotation failure is the worst-case shape — operator thinks they rotated, old key is still encrypt-with, KMS rotation logs say done.

**Fix:**
- Add an integration assertion at end of `reencrypt_all`: re-read 1 random row's raw payload, base64-decode the Fernet token, verify the `time_low` field's first 4 bytes (Fernet HMAC prefix derived from primary key) matches `cipher.key_fingerprint()`. Or simpler: assert `Fernet(primary_key).decrypt(raw_token)` succeeds and `Fernet(old_key).decrypt(raw_token)` raises.
- Add the `_encrypt_request_json` + `_inner` symbols to the library's public API with a stability guarantee, OR expose a `rotate(cipher)` method on `EncryptedCheckpointStore`.

## MEDIUM

### A8-M-01 — `_namespace_key()` returns signed int64 but XOR truncates to low 32 bits → 32-bit namespace collision space
**File:** `examples/production/durable_postgres/lock.py`, lines 83, 120

```python
return int.from_bytes(digest[0:8], "big", signed=True)  # 64-bit namespace
...
ns_bits = self._namespace & 0xFFFFFFFF  # low 32 bits only
ns_k1 = k1 ^ ns_bits
```

The namespace digest is 64 bits but only XOR'd into key1 (32 bits). key2 is never namespaced. Effective namespace cardinality: 2^32 ≈ 4B. Birthday collision at √(2·2^32) ≈ 93K namespaces. Real-world co-residency with pg_boss / hangfire / a 2nd durable-checkpoints deployment in the same DB cluster → collision risk is small but the doc oversells: "64-bit namespace" appears in lock.py line 14, but actual entropy is 32 bits.

**Impact:** two tenants with colliding 32-bit hashes share key-space → tenant A's `acquire(run_id="abc")` blocks tenant B's `acquire(run_id="abc")` (or worse, tenant B steals when tenant A releases).

**Fix:** XOR full 64 bits — `ns_low = ns & 0xFFFFFFFF`, `ns_high = (ns >> 32) & 0xFFFFFFFF`, `ns_k1 = k1 ^ ns_low`, `ns_k2 = k2 ^ ns_high`. Then update tests to check both keys move.

### A8-M-02 — `try/except (ValueError, Exception)` in cipher.py catches the same exception twice; `Exception` swallows everything including `KeyboardInterrupt`-sibling errors
**File:** `examples/production/durable_postgres/cipher.py`, line 44

```python
except (ValueError, Exception) as exc:
    raise ValueError(f"invalid Fernet key: {exc}") from exc
```

`ValueError` is a subclass of `Exception`; the tuple is redundant. More importantly, broad `Exception` catches `MemoryError`, `SystemError`, `RecursionError` — re-raising them as `ValueError("invalid Fernet key: ...")` masks the real failure mode. An OOM during Fernet construction would now look like "bad key shape" — operator wastes time regenerating keys when the real fix is more memory.

**Fix:** `except (ValueError, TypeError, binascii.Error) as exc:` — name the actual failure shapes (Fernet rejects bad keys with `ValueError`; base64 decode can throw `binascii.Error`; wrong type throws `TypeError`).

### A8-M-03 — `daemon.HealthcheckServer._handle_inner` writes 500-error response inside the `except Exception` block that catches its own writer errors — double-write to broken socket
**File:** `examples/production/durable_postgres/daemon.py`, lines 217–224

```python
except Exception:
    try:
        writer.write(b"HTTP/1.1 500 Internal Server Error\r\n\r\n")
        await writer.drain()
    except Exception:
        pass
```

If the `try:` block already wrote a partial response (e.g. status line and Content-Length, then crashed before body), the `500 Internal Server Error\r\n\r\n` is appended to the same TCP stream → mixed response, client sees garbage. The plan-review N-L-05 noted this; the fix called for a `wrote_response` bool guard. The implementation did NOT add the guard.

**Impact:** healthcheck client gets non-parseable HTTP → docker reports container unhealthy → false-positive restart cascade.

**Fix:** track `wrote_response = False` flag; only emit 500 if `not wrote_response`.

### A8-M-04 — `_handle_inner` does not validate HTTP version, accepts any trailing garbage on request line
**File:** `examples/production/durable_postgres/daemon.py`, lines 191, 204–206

```python
method_path = request_line.decode("ascii", errors="replace").split()
...
if (len(method_path) >= 2
        and method_path[0] == "GET"
        and method_path[1] == "/health"):
```

`>= 2` and only checks first two tokens. `GET /health junk junk junk\r\n` passes. `GET /health\r\nGET /health\r\n` after a tampered readline (`errors="replace"`) survives. Not exploitable today — there's no smuggling vector because the daemon doesn't proxy — but loosens the parser unnecessarily. Plan-review N-L-04 flagged this; not fixed in ship.

**Fix:** `len(method_path) == 3 and method_path[2].startswith("HTTP/")`.

### A8-M-05 — `daemon.workflow_factory` uses `assert` for workflow-class allowlist enforcement; `python -O` strips it
**File:** `examples/production/durable_postgres/daemon.py`, line 291

```python
def workflow_factory(workflow_class: str) -> DurableWorkflow:
    assert workflow_class.endswith("ClinicalTrialEligibilityDurableWorkflow")
```

The `assert` is the only barrier between the scheduler's `list_paused` results (which read `workflow_class` from the DB column) and instantiation. If an attacker writes a row with `workflow_class="x.Y.Evil"` to the DB (requires DB write, which the daemon role has) and `python -O` is set in any future Dockerfile rev → assert is a no-op → ClinicalTrial workflow constructed for arbitrary class string. Currently the factory hard-codes `ClinicalTrialEligibilityDurableWorkflow` regardless, so this is latent — but a future "match on workflow_class" dispatch table inherits the broken gate.

The `ENTRYPOINT ["sh", "-c", "ulimit -c 0 && exec python -m daemon"]` is `python` not `python -O` today. Still: asserts for security gates is convention-level error compounding (same shape as M-PC-1 / H-IND-1 noted in CLAUDE.md).

**Fix:** raise `ValueError` explicitly, not `assert`. Or use a frozenset allowlist with explicit raise.

### A8-M-06 — Container builds and runs are full root inside Postgres image (no hardening on `postgres` service)
**File:** `examples/production/durable_postgres/docker-compose.yml`, lines 13–30

The `postgres` service has no `cap_drop`, no `read_only`, no `security_opt`, no `ulimits`. Just `image: postgres:16-alpine` + healthcheck. Compared to the scheduler service (lines 32–56) which has the full hardening stack, postgres is unprotected.

**Impact:** if a Postgres CVE allows code execution (recent example: CVE-2024-10977 client-side), the postgres container runs as root with all caps → trivial breakout via `/var/run/docker.sock` (not mounted here, good) or `/proc/sys` writes (writable rootfs). The README hardening table (line 119) implicitly applies to the scheduler only; nowhere is "postgres also needs hardening" stated.

**Fix:** apply `cap_drop: [ALL]` + `cap_add: [CHOWN, DAC_OVERRIDE, FOWNER, SETGID, SETUID]` (postgres minimum) + `security_opt: [no-new-privileges:true]` + `ulimits.core: 0` to the postgres service. Pin the postgres image by digest too (matches Dockerfile pattern).

### A8-M-07 — `postgres` image is `postgres:16-alpine` tag, not digest-pinned; ships floating tag
**File:** `examples/production/durable_postgres/docker-compose.yml`, line 15

```yaml
image: postgres:16-alpine
```

Inconsistent with the Dockerfile's `@sha256:` pin. Operator pulls today, attacker MitMs Docker Hub tomorrow, deploy ships a different image.

**Fix:** `image: postgres:16-alpine@sha256:<digest>` — and document the refresh next to the Python base image digest in README.

### A8-M-08 — `tests/conftest.py` `fresh_checkpoints_table` executes `schema.sql` raw via `conn.execute()`; injection-vulnerable if `schema.sql` is ever templated, AND DROPs the production table when run against a real DSN
**File:** `examples/production/durable_postgres/tests/conftest.py`, lines 44–53

```python
async def fresh_checkpoints_table(pg_pool):
    async with pg_pool.acquire() as conn:
        await conn.execute("DROP TABLE IF EXISTS checkpoints CASCADE;")
```

A developer who points `POSTGRES_DSN` at staging (or worse, production via copy-paste) and runs `pytest examples/production/durable_postgres/` **wipes the production checkpoints table** — every paused run is lost. No `if "test" in dsn` guard, no confirmation prompt, no schema isolation.

The library has H-DUR-3 (path-traversal containment) closed; this is the equivalent at the DB level — no enforcement that the test runs against a test DB.

**Impact:** human error → total data loss on the durability surface. The whole point of the durable subpackage is to survive process death; a careless `pytest` invocation against the wrong DSN destroys that.

**Fix:**
- Skip if `POSTGRES_DSN` doesn't contain `test` substring OR isn't in an allowlist of `localhost`/`127.0.0.1`/explicit test hosts.
- Better: use a `pytest-postgresql` ephemeral instance.
- At minimum: raise loud error if DB `checkpoints` table already has > 0 rows that don't match a test-data prefix (`tst-`, `t1-`, etc.).

### A8-M-09 — `_serialize` uses `ensure_ascii=False` → multi-byte UTF-8 bytes hit BYTEA storage; legitimate but combined with `_deserialize` `errors=` absent → silent corruption on partial reads
**File:** `examples/production/durable_postgres/store.py`, lines 257, 262

```python
return json.dumps(body, ensure_ascii=False).encode("utf-8")  # write
...
body = json.loads(bytes(row["payload"]).decode("utf-8"))      # read
```

If a future migration ever copies BYTEA → TEXT column (encoding-aware) and the column collation doesn't match `client_encoding`, the round-trip silently mojibakes. The `CheckpointCorrupt` raise at line 264 handles JSON parse errors but not silent corruption of valid-JSON-with-wrong-bytes.

Lower priority because BYTEA is opaque today, but the comment at line 7-8 ("payload column is BYTEA NOT NULL ... encrypted ciphertext") sets up future readers to expect bytes. The `ensure_ascii=False` + `errors="strict"` default-decode pair is the right default but should be documented.

**Fix:** explicit `errors="strict"` on decode (current default but clearer), comment noting BYTEA is the canonical type forever.

### A8-L-08 — `.env.example` line 12 shows `POSTGRES_DSN=postgresql://daemon:CHANGEME@localhost:5432/durable` — `CHANGEME` placeholder pattern, no warning that compose overrides this
**File:** `examples/production/durable_postgres/.env.example`, line 12
