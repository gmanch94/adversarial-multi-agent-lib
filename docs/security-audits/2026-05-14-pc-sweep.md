## MEDIUM

### M-PC-1 — `_extract_veto` initial split is not line-anchored (convention-level gap across all 4 veto-using workflows)

**Files / lines:**
- `src/adv_multi_agent/pc/workflows/claims_reserve.py:435-437`
- `src/adv_multi_agent/pc/workflows/coverage_decision.py:358-360`
- `src/adv_multi_agent/pc/workflows/environmental_impairment.py:388-390`
- `src/adv_multi_agent/pc/workflows/gig_platform_liability.py:401-403`
- (Pre-existing parity gap also present in `src/adv_multi_agent/retail/workflows/recall_scope.py:368-370` — inherited, not newly introduced.)

**Attack vector.** All 4 clones do:
```python
if "REVIEWER VETO:" not in critique:
    return None
section = critique.split("REVIEWER VETO:", 1)[1]
```
Each workflow's `_RESERVE_REVIEW_CRITERIA` / `_COVERAGE_REVIEW_CRITERIA` / etc. embeds the literal string `REVIEWER VETO:` 4-6 times inside the criteria block that is sent to the reviewer. A reviewer model that quotes the criteria back in its critique prose — e.g. *"Per the REVIEWER VETO: criteria above, I evaluated the life-safety signal..."* — causes `split("REVIEWER VETO:", 1)` to anchor at the **first quoted occurrence**, not the trailing directive line.

After the bad anchor:
- If the prose continuation lines do not contain `Overall`, `Key issues`, `#`, or a sibling uppercase-colon header, the parser collects narrative as the "veto directive" → **false-positive veto** (workflow halts on a non-veto critique).
- If the prose is followed by a sibling-uppercase header before reaching the real `REVIEWER VETO: None` / actual directive line, the parser captures the narrative span; the real terminal directive is never read → **false-negative on a real veto** OR **false-positive on a non-veto** depending on prose content.

**Why this is the same shape as the closed M1.** The 2026-05-13 audit closed exactly this pattern for `extract_flags` by switching to a line-anchored regex (`extract_flags` uses `re.search(rf"(?m)^\s*{re.escape(header)}", critique)`). The veto parser was not similarly hardened — and the veto is the **strongest** convergence gate in the system (an `_extract_veto` false-positive forces an irreversible halt with `metadata['vetoed']=True` written to audit; a false-negative passes a vetoed plan to convergence).

**Impact.** Audit-trail integrity (workflow records a synthetic veto reason composed of paraphrased criteria text) + safety-property regression (a real veto on a life-safety / SOX-restatement / coverage-denial signal can be silently dropped). Severity MEDIUM rather than HIGH only because (a) the reviewer prompt instructs verbatim format and a well-behaved reviewer outputs the marker exactly once, and (b) the audit-trail `wiki.add_feedback` write happens before `_extract_veto` so the raw critique is captured even if veto-parse mis-fires.

**Remediation.** Apply the same M1 line-anchored regex pattern to the veto parser. Replace:
```python
if "REVIEWER VETO:" not in critique:
    return None
section = critique.split("REVIEWER VETO:", 1)[1]
```
with:
```python
match = re.search(r"(?m)^\s*REVIEWER VETO:\s*(.*)$", critique)
if match is None:
    return None
# resume with section = the post-match text starting from match.end()
```
Apply consistently across all 4 clones + the retail `recall_scope.py` reference. This should live as a shared helper in `core/_internal.py` (`extract_veto_directive`) to prevent the next workflow from re-introducing the gap — the current copy-paste pattern is exactly the "convention-level error compounding" failure mode called out in CLAUDE.md.

### L-PC-1 — `_extract_veto` clones diverge from each other only via docstrings; logic itself is byte-identical → consolidation opportunity (defence-in-depth)

**Files:** all 4 clones listed in M-PC-1.

**Observation.** The 4 implementations of `_extract_veto` are line-for-line identical apart from the docstring "Mirrors..." attribution. Each clone is a future place where a hardening fix can fail to land. Even after M-PC-1 is fixed, the convention will re-emerge in the next veto-using workflow unless the function is hoisted to `core/_internal.py` or `core/workflow.py`. Cost is ~30 LOC saved + 1 fix-once invariant.

**Severity:** LOW (no live exploit) but high-leverage refactor.

### L-PC-2 — Veto stop-list captures legitimate veto directives that begin with "Overall" / "Key issues" / a `#` markdown header

**Files / lines:** same 4 clones; condition at e.g. `claims_reserve.py:459`:
```python
if lower.startswith(("overall", "key issues", "#")) or sibling_header:
    break
```

**Attack vector / false-negative.** If the reviewer phrases the verbatim directive naturally — e.g. *"REVIEWER VETO: Overall, escalate to senior actuary; reserve below comparable median in plaintiff-friendly venue"* — the multi-line collection breaks the moment a continuation line begins with `Overall`. Single-line vetoes are fine; multi-line vetoes that happen to start a continuation with one of those words drop the trailing content.

**Severity LOW** because (a) the criteria block instructs verbatim directive format starting with an action verb, and (b) the first line is always captured regardless of prefix (the stop-list only applies to continuation lines, `idx > 0`).

**Remediation.** Document this constraint in the criteria prompt template ("Do not begin a continuation line with `Overall`, `Key issues`, or a markdown header — these terminate the veto block"); or relax the stop-list to require the markers to be at column 0 with a markdown-section followup. Lowest-effort fix: just add the note to the criteria block.

### L-PC-3 — `audit_complaint_history` and similar long-form free-text fields in PC Request dataclasses are length-capped only via `sanitize_for_prompt(request_text, max_chars=6000)` on the **concatenated** prompt, not per-field

**Files:** all 7 `Request.to_prompt_text()` methods join free-text fields with newlines and then the workflow caps the concatenated result at 6000 chars. The retail `LoyaltyOfferRequest` capped its `list[str]` fields per-entry at 200 chars × 64 entries (`_MAX_ATTRIBUTE_ENTRIES`). PC has no list fields, but the per-field caps are implicit (a single 6000-char `regulatory_exposure` would consume the entire budget and truncate later fields).

**Attack vector.** A caller-supplied `comparable_cases` field of 50,000 chars would push all later fields (`regulatory_exposure`, `current_reserve_proposal`) out of the truncation window. The workflow then has reviewer scoring blind to those later fields, plausibly affecting reserve-adequacy or veto-criterion evaluation.

**Severity LOW** — this is a denial-of-fidelity / silent-truncation risk, not a code-execution or audit-integrity break. The library is teaching-grade by stated posture (`PRODUCTION_GAPS`).

**Remediation.** Add per-field caps in `to_prompt_text` (e.g. `f"Comparable cases: {self.comparable_cases[:1500]}"`) for the 1-2 longest-expected fields per request. Alternative: document the implicit truncation budget in each Request docstring.

### L-PC-4 — Skill template inputs are interpolated as suffix to descriptive labels, but `_PartialFormat` passthrough means an input value containing `{xyz}` is rendered as a literal token in the prompt

**Files:** all 29 skill templates in `src/adv_multi_agent/pc/skills/templates/`.

**Attack vector.** Per `registry.py` `Skill.render` uses `format_map(_PartialFormat(...))` which passes through unknown `{xyz}` substrings unchanged. An input value containing e.g. `{system_override}` would render as literal `{system_override}` in the prompt — not a code-injection, but a prompt-smuggling vector where the LLM may interpret the brace-syntax as authoritative. Combined with `sanitize_for_prompt` cap (8192 per input from registry) but no brace-stripping.

**Severity LOW** — LLMs do not natively treat `{xyz}` as control syntax; the risk is purely social-engineering against the model.

**Remediation.** Either strip / escape `{` and `}` in input values inside `Skill.render`, or document that callers must not pass brace-laden content. Low priority.

### L-PC-5 — `_format_flag_section` re-injection wraps each flag in `sanitize_for_prompt(f, max_chars=500)`, but the per-flag count is bounded only by `_MAX_FLAGS_PER_HEADER=64` in `extract_flags` — three flag headers × 64 × 500 = 96 KB worst-case injected back into the next-round prompt

**Files:** all 5 triple-flag workflows' `_format_flag_section`.

**Attack vector.** A reviewer that emits 64 max-length flags per header (defence-in-depth cap from L2) re-injects up to ~96 KB of attacker-influenced content into the next round's executor prompt, potentially exceeding model context windows or amplifying prompt-injection surface.

**Severity LOW** — practically the reviewer critique itself is capped at 4000 chars upstream, so it cannot physically emit 64 max-length flags per header in one critique. Worst real-case is bounded by the 4000-char `critique` upstream cap → maybe 6-8 flags per header in practice.

**Remediation.** Add an explicit `len(flags) > 16` defensive truncation in `_format_flag_section` if a tighter bound is desired. Optional.
