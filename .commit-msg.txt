fix(security): MEDIUM hardening M2 M4 M5

Targets three MEDIUM findings from docs/security-audits/2026-05-13.md.

M2 (unbounded claim-line count -> ledger explosion): cap claims
registered per round at 200 in AutoReviewLoop._extract_and_register_claims.
Adversarial executor emitting 10k "## Claims" lines no longer fills
the disk via per-add atomic save. Legitimate rounds emit < 50.

M4 (silent corrupt-file load erases audit trail): ClaimLedger._load
and ResearchWiki._load now emit UserWarning before starting fresh.
Operator can see corruption/tampering instead of audit trail vanishing
silently. Workflow still continues to avoid hard-failing the run.

M5 (no type guard on Anthropic block.text): _extract_anthropic_text
defensively coerces non-string text values via str() in case of
future SDK changes. Prevents latent TypeError on SDK upgrade.

Skipped from this batch:
- M1 (wiki approve_improvement caller-identity gate): architectural,
  needs design discussion. Python class-method protection is
  convention-only; meaningful enforcement requires a session token
  or capability pattern.
- M3 (async lock on _save): false positive. _save is sync, so
  Python's GIL/coroutine semantics already serialize. No race.
- M9 (EffortLevel for OpenAI reviewer): OpenAI chat.completions has
  no direct effort knob; documentation fix more than code fix.

Tests: +2 unit tests (M2 claim cap, M4 ledger warning). 218 -> 220
passing. No new lint/mypy errors vs baseline.

Stacks on fix/security-hardening-mcp-dos (PR #6).
